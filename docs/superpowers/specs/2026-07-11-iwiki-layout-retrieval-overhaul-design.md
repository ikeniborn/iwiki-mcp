# iwiki layout + retrieval overhaul — design

**Date:** 2026-07-11
**Branch:** `dev-iwiki-layout-retrieval` (off `master`, v0.5.1) → PR into `master`
**Status:** approved design, pre-implementation

## Context

Two-level hierarchical retrieval (Unit B, v0.5.0) already stores the frontmatter
`description` as a disjoint `kind="summary"` vector and each section as a clean
`kind="section"` vector, then retrieves via seed-summary → wiki-graph expand →
section rank (`retrieval.py`, `engine/hier.py`). The retrieval *principle* is
correct and unchanged by this work.

This overhaul bundles six related changes (A–F) into one release. The unifying
idea is a single **page identity rule**, then storage-layout and precision work
built on it.

## Page identity rule (foundation for C/D)

**A page's identity is its domain-relative path**, already true for the index
`file` field and for `engine/links.parse_links` keys (`slug` / `slug#heading`).
This overhaul makes the article `type` the **first path segment** of that
identity: a page lives at `<base>/<domain>/<type>/<slug>.md` and its identity is
`<type>/<slug>`. The frontmatter `type` MUST equal the first path segment.
Everything else (links, index paths, read/write addressing) follows from this
one rule.

## A. Fix AUTHORING_RULES text (`resources.py`)

The rules string is stale under the two-level model and the new layout.

- Lines 27-29: remove the false claim that `description` is "embedded as each
  section's context prefix". Replace with: `description` is the article summary,
  indexed as its own **summary-level vector that seeds retrieval** (two-level:
  summary seed → graph-expanded pool → section vectors ranked inside it), **not**
  prefixed onto section vectors. Keep the "write it rich with `Covers:` / `Terms:`"
  guidance, reframed as "so the summary matches broad queries".
- Update the link-format guidance (line 19) to the new identity: cross-links use
  `[Heading](<type>/<slug>.md#heading)` within the domain.
- Update the reserved-file note (line 38): `index.md` / `log.md` are **export-only**
  OKF artifacts (generated by `wiki_export_okf`), no longer refreshed on every write.
- No behavior change here beyond text; it is authored-agent guidance surfaced as
  the `iwiki://authoring-rules` resource.

## B. Retrieval (verified — no change)

`retrieval.py` + `engine/hier.py` already implement the intended flow:
seed `kind="summary"` by cosine → `expand_graph` undirected wiki-graph BFS →
`rank_sections` cosine over pooled `kind="section"` → hybrid merge. No change
except what C (adjacency path keys) and F (write mode) require below.

## C. Move `index.jsonl` / `log.jsonl` to the domain root

Drop the hidden `.iwiki/` subdir; the store and log live at the domain root.

- **`base.py`** — `index_path` → `<base>/<domain>/index.jsonl`;
  `log_path` → `<base>/<domain>/log.jsonl`.
- **`okf.py`** — `_read_log` reads `<domain>/log.jsonl` (drop the `.iwiki/`
  hardcode at line 128).
- **rglob excludes** — `indexer.py:36`, `server.py:206-208,821-823`,
  `okf._page_slugs:121` currently skip a `.iwiki` path segment. The relocated
  `*.jsonl` files are not `*.md`, so `rglob("*.md")` never picks them up; remove
  the now-dead `.iwiki` guard for clarity (keep the `RESERVED_OKF` exclusion).
- **`wiki_create_domain`** — `server.py:642` runs
  `os.makedirs(dom_path / ".iwiki", exist_ok=True)`. Drop it so new domains do not
  recreate an empty `.iwiki/`; the store/log files are created lazily at the domain
  root on first index/log write.
- **Migration** — a one-time move of each domain's existing `.iwiki/index.jsonl`
  and `.iwiki/log.jsonl` to the domain root, then remove the empty `.iwiki/` dir.
  Handled by `wiki_migrate_okf` (see D) so a single migrate pass fixes both layout
  and store location per domain.

## D. Type-grouping `<domain>/<type>/<slug>.md` (core)

Page identity = `<type>/<slug>`, type = first path segment == frontmatter `type`.

- **`server.py` write tools** (`wiki_write_page`, `wiki_apply_okf`) — resolve the
  on-disk path from identity: if `slug` has no `/` and a `type` is given, the
  server prefixes it to `<type>/<slug>`; if `slug` already carries a first
  segment, validate it equals the resolved frontmatter `type` (reject mismatch
  with an `{error, hint}`). The returned page id is `<type>/<slug>`.
- **Addressing** — `wiki_read_page` / `wiki_update_page` / `wiki_delete_page` take
  the full `<type>/<slug>` identity. `_slug_parts` already supports nested paths;
  its traversal guards (`.`/`..`/absolute/drive) already cover the extra segment.
- **Type as a path segment** — validate the type value is a safe single segment
  (lowercase open vocab via `fm.normalize_type`, plus reject `.`-prefix, `/`, `..`).
- **`engine/hier.py` `_adjacency` (BREAKS)** — currently flat `os.listdir(root)`,
  top-level `*.md` only. Rewrite to `rglob("*.md")` keyed by domain-relative posix
  path (excluding `RESERVED_OKF`; the `*.jsonl` store files are not `*.md`); link targets are
  domain-relative `<type>/<slug>.md`. This restores cross-type graph edges — the
  load-bearing fix, without which graph expansion silently loses links.
- **Links** — authored/rewritten as `[Heading](<type>/<slug>.md#heading)`.
  `engine/links.parse_links` already normalizes a markdown target to its
  domain-relative slug key, so nested targets work without parser changes.
  `engine/lint.py` cross-reference resolution resolves `<type>/<slug>` identities.
- **Type change** — NOT via `wiki_update_page` (it edits one `##` section body and
  never moves files). A type change moves the file to the new type dir and
  rewrites inbound links; performed by `wiki_apply_okf` and the migration.
- **Migration (`wiki_migrate_okf`)** — for each flat `<slug>.md`, read its
  frontmatter `type`, move it to `<type>/<slug>.md`, rewrite intra-domain links
  from `<slug>.md` to `<type>/<slug>.md`, and re-index. Combined with the C store
  relocation in the same pass. Intra-domain links only (cross-domain links are out
  of scope in v1, matching current authoring rules).

## E. `index.md` / `log.md` — export-only

- **`server.py`** — remove the `okf.refresh_artifacts` call from every write path
  (`wiki_write_page`, `wiki_update_page`, `wiki_delete_page`, `wiki_migrate_okf`,
  `wiki_apply_okf`: lines ~429, 530, 587, 866, 945). Keep it **only** in
  `wiki_export_okf`, which already calls it (line 976).
- **Keep** the `RESERVED_OKF` (`index.md` / `log.md`) exclusion in `indexer.py`,
  `engine/grep.py`, `engine/lint.py`, `okf._page_slugs`, and the write-guard that
  rejects the reserved slugs, so an exported `index.md` never enters the vectors
  and can't be authored by a tool.
- Net effect: within a project the domain holds only real pages + the two `*.jsonl`
  store files; `index.md`/`log.md` are produced on demand by `wiki_export_okf` for
  native-OKF export.

## F. Precise write-target search mode

A dedicated precise mode so a write knows the exact article + section to upsert.

- **`engine/hier.py` / `retrieval.py`** — a `locate_target(cfg, base, domain,
  query, heading=None)` path: raise the seed threshold to `write_seed_threshold`,
  and when a `heading` hint is given require an **exact (case-insensitive) heading
  match** in `rank_sections`. Returns the single best `{domain, file, heading,
  score}` plus `exists: bool` (whether a matching article+section already exists).
- **`server.py`** — surface via `wiki_search(..., intent="write", heading=None)`:
  `intent="write"` selects the precise branch; default `intent="read"` keeps
  today's hybrid behavior. The write tools call the same `locate_target` helper
  internally to decide create-new-section vs update-existing.
- **`engine/config.py`** — add `write_seed_threshold` (default above
  `seed_threshold`). No change to read-path defaults.

## Migration / index versioning

One migration pass per domain (`wiki_migrate_okf`) performs, idempotently:
1. Move `.iwiki/index.jsonl` + `.iwiki/log.jsonl` to the domain root; drop `.iwiki/`.
2. Move each flat `<slug>.md` to `<type>/<slug>.md`; rewrite intra-domain links.
3. Re-index (the store `file` paths become `<type>/<slug>`).

No embedding re-cost beyond re-index of moved pages (hashes unchanged where text
is unchanged; only `file` keys move). Documented for operators as a one-time
`wiki_migrate_okf` per domain after upgrade.

## Success criteria

- `AUTHORING_RULES` text matches the two-level model, the `<type>/<slug>` link
  format, and export-only artifacts (resource fetch shows the corrected text).
- A page written with `type=architecture, slug=retrieval` lands at
  `architecture/retrieval.md`; its identity, index `file`, and links are
  `architecture/retrieval`. A `type`/first-segment mismatch is rejected.
- `hier._adjacency` builds cross-type edges (a link from `guide/a.md` to
  `api/b.md` yields an undirected edge); graph expansion reaches it.
- After `wiki_migrate_okf`, existing flat domains are re-laid-out under type dirs,
  stores live at the domain root, links are rewritten, and search still returns
  the expected articles+sections.
- No write path writes `index.md`/`log.md`; `wiki_export_okf` still does.
- `wiki_search(intent="write", heading=...)` returns a single precise
  article+section with an `exists` flag; write tools use it for create-vs-update.
- `uv run pytest -q` green; `uv run flake8 src tests` clean; live smoke on the
  `iwiki-mcp` domain after migration.

## Tests

- `resources`: authored rules contain the new description/link/export wording and
  none of the stale prefix claim.
- `base`: `index_path`/`log_path` point at the domain root.
- `server` write/read/update/delete: `type/slug` addressing round-trips; first-
  segment/type mismatch rejected; path-traversal guards still reject `..`/absolute.
- `hier`: `_adjacency` over a nested (type-dir) fixture builds the correct
  undirected edges; expansion crosses type dirs.
- `okf`/`server`: no write path emits `index.md`/`log.md`; `wiki_export_okf` emits
  both; `RESERVED_OKF` still excluded from index/grep/lint.
- `migrate`: a flat domain with `.iwiki/` migrates to type dirs + root store +
  rewritten links, idempotently (second run is a no-op); no empty `.iwiki/` remains.
- `store`/`migrate`: a `kind="summary"` vector persists for a page after migration
  (guards the requirement that `description` stays a separate summary vector).
- `retrieval`/`hier`: `locate_target` honors `write_seed_threshold` and exact-
  heading match; `exists` correct; `wiki_search(intent="write")` shape.

- Auto-move on `wiki_update_page` type edits (type change is apply/migrate only).
